# รายงานจำลองอนาคต TH-AI Passport

## ความท้าทายของโครงสร้างพื้นฐาน AI แห่งชาติ และปัญหาการใช้งานจริงของประชาชน

โครงการ TH-AI Passport ถูกออกแบบให้เป็นนโยบายระดับประเทศ เพื่อช่วยให้คนไทยเข้าถึง AI ได้มากขึ้น แต่จากภาพจำลองอนาคต โครงการนี้กลับสะท้อนปัญหาสำคัญหลายด้านพร้อมกัน

ด้านหนึ่ง รัฐพยายามสร้างภาพของความเปิดกว้างทางเทคโนโลยี เช่น การใช้โมเดล AI แบบโอเพนซอร์สภายใต้สัญญาอนุญาต Apache 2.0 แต่อีกด้านหนึ่ง โครงการกลับเผชิญปัญหาเรื่องความเชื่อมั่นในการจัดซื้อ ความไม่ชัดเจนด้านข้อมูลส่วนบุคคล ความเสี่ยงเรื่องการพึ่งพาผู้ให้บริการต่างชาติ และอัตราการใช้งานต่อเนื่องของประชาชนที่ต่ำ

พูดให้เข้าใจง่ายคือ โครงการนี้ไม่ได้มีปัญหาแค่ว่า “มี AI ให้ใช้หรือไม่” แต่มีปัญหาลึกกว่านั้น คือ “ใครควบคุมระบบนี้จริง” “ข้อมูลประชาชนถูกจัดการอย่างไร” และ “ประชาชนใช้แล้วเกิดประโยชน์จริงหรือไม่”

---

# 1. ธรรมาภิบาลหลุดโฟกัส: จุดเสี่ยงสำคัญของโครงการ

รายงานนี้ชี้ให้เห็น 3 จุดเสี่ยงใหญ่ที่ทำให้โครงการ TH-AI Passport ถูกตั้งคำถามอย่างหนัก ได้แก่ ความเชื่อมั่นด้านการจัดซื้อ ความเป็นส่วนตัวของข้อมูล และอธิปไตยทางเทคโนโลยี

---

## 1.1 ความเชื่อมั่นด้านการจัดซื้อเริ่มพังลง

สำนักงานการตรวจเงินแผ่นดิน หรือ SAO ได้ออกรายงานตรวจสอบเบื้องต้นเมื่อวันที่ 5 เมษายน 2027 โดยระบุว่า เอกสารประกวดราคาของโครงการ TH-AI Passport หมายเลข TOR-2026-047 ไม่ผ่านเกณฑ์ตรวจสอบความสมบูรณ์ของการจัดซื้อระบบดิจิทัล

ปัญหาหลักอยู่ที่ 2 เรื่องสำคัญ คือ

1. **ข้อมูลและระบบสามารถย้ายออกได้จริงหรือไม่**
2. **สถาปัตยกรรมทางเทคนิคถูกเปิดเผยชัดเจนพอหรือไม่**

ข้อค้นพบนี้ทำให้ข้อกังวลของสังคมไม่ได้เป็นเพียงข่าวลือหรือการคาดเดาอีกต่อไป แต่กลายเป็นประเด็นที่มีหน่วยงานตรวจสอบของรัฐเข้ามายืนยัน

ขณะเดียวกัน ฝ่ายค้านและสื่อยังตั้งคำถามเรื่องงบประมาณของโครงการว่า หากเป้าหมายคือการแจกสิทธิ์ใช้งาน AI ฟรีให้ประชาชน เหตุใดจึงไม่มีข้อมูลที่ชัดเจนว่า มีคนใช้งานจริงเท่าไร ใครได้รับประโยชน์จริง และผลลัพธ์ที่เกิดขึ้นคุ้มค่ากับงบประมาณหรือไม่

ประเด็นที่รุนแรงกว่านั้นคือ เอกสาร TOR ไม่ได้ถูกเผยแพร่ล่วงหน้าอย่างถูกต้องตามข้อกำหนดของกฎหมายจัดซื้อจัดจ้าง ทำให้องค์กรภาคประชาชนด้านเทคโนโลยีต้องเปิดแผนติดตามพิเศษ และระบุว่าสถานะความสมบูรณ์ของเอกสารจัดซื้อยังอยู่ในระดับเสี่ยงสูง

พูดแบบง่ายที่สุดคือ โครงการนี้เริ่มมีปัญหาตั้งแต่ขั้นตอนการจัดซื้อ เพราะสังคมยังไม่เห็นข้อมูลสำคัญที่ควรเปิดเผย

---

## 1.2 ความเป็นส่วนตัวของข้อมูลยังไม่มีคำตอบชัดเจน

โครงการ TH-AI Passport เกี่ยวข้องกับข้อมูลผู้ใช้งานจำนวนมาก โดยเฉพาะข้อมูลยืนยันตัวตน การเข้าใช้งาน และประวัติการใช้ระบบ AI ดังนั้น โครงการลักษณะนี้ควรมีการประเมินผลกระทบด้านความเป็นส่วนตัว หรือ PIA อย่างชัดเจน

แม้กฎหมายคุ้มครองข้อมูลส่วนบุคคลจะกำหนดให้ระบบที่มีความเสี่ยงสูงต้องจัดทำและเปิดเผยการประเมินผลกระทบด้านความเป็นส่วนตัว แต่ในภาพจำลองนี้ โครงการ TH-AI Passport ยังไม่มีการเผยแพร่เอกสาร PIA ที่ตรวจสอบได้บนเว็บไซต์ของหน่วยงานกำกับดูแล

สิ่งที่ทำให้สังคมยิ่งตั้งคำถามคือ รัฐมนตรีกระทรวงดิจิทัลฯ เคยกดถูกใจโพสต์ที่เรียกร้องให้เปิดเผย PIA แต่หลังจากนั้นกลับไม่มีการดำเนินการที่ชัดเจนตามมา

สำหรับภาคประชาชน สิ่งนี้ถูกมองว่าเป็น “ความเงียบของสถาบัน” เพราะรัฐดูเหมือนรับรู้ปัญหา แต่ไม่ได้ให้คำตอบที่เป็นรูปธรรม

คำถามสำคัญที่ยังไม่มีคำตอบ ได้แก่

* ข้อมูล log การเข้าใช้งานจะถูกเก็บไว้นานแค่ไหน
* ใครมีสิทธิ์เข้าถึงข้อมูลเหล่านี้
* ข้อมูลถูกส่งไปยังต่างประเทศหรือไม่
* หากมีการตรวจสอบย้อนหลัง ใครเป็นผู้รับผิดชอบ
* ผู้ใช้สามารถรู้ได้หรือไม่ว่าข้อมูลของตนถูกนำไปใช้อย่างไร

ประเด็นนี้จึงไม่ใช่แค่เรื่องเทคนิค แต่เป็นเรื่องความไว้วางใจระหว่างรัฐกับประชาชน

---

## 1.3 ความเสี่ยงเรื่องการพึ่งพาผู้ให้บริการรายใหญ่

อีกหนึ่งประเด็นใหญ่คือความเสี่ยงที่โครงการจะถูกผูกติดกับผู้ให้บริการเทคโนโลยีรายใดรายหนึ่ง หรือที่เรียกว่า **vendor lock-in**

นักพัฒนา AI สตาร์ทอัป และนักวิจัย LLM ในไทยต่างกังวลว่า แม้โครงการจะพูดถึงความสามารถในการย้ายโมเดลและข้อมูลกลับเข้าสู่ระบบที่รัฐควบคุมได้ แต่ในทางปฏิบัติกลับไม่มีรายละเอียดทางเทคนิคที่เปิดเผยชัดเจน

สิ่งที่ยังไม่ถูกเปิดเผย ได้แก่

* แผนผังสถาปัตยกรรมของระบบ
* วิธีจัดสรรเครดิต API
* ข้อกำหนดด้านคุณภาพบริการ หรือ SLA
* วิธี route โมเดลระหว่างผู้ให้บริการ
* ตำแหน่งที่เก็บ prompt log
* รายชื่อผู้ประมวลผลข้อมูลรายย่อย หรือ sub-processor

นักพัฒนาจึงเรียกร้องให้มีเอกสาร technical whitepaper และ dashboard สำหรับดูการใช้เครดิต API แต่จนถึงช่วงเวลาจำลอง สิ่งเหล่านี้ยังมีอยู่เพียงในวงสนทนา ไม่ได้ถูกเผยแพร่อย่างเป็นทางการ

สรุปง่าย ๆ คือ แม้โครงการจะพูดเรื่องอธิปไตยทางเทคโนโลยี แต่หากไม่เปิดเผยโครงสร้างระบบ ก็ยากที่สังคมจะเชื่อว่ารัฐควบคุมระบบนี้ได้จริง

# 2. ความย้อนแย้งระหว่าง “โอเพนซอร์ส” กับ “อธิปไตยทางเทคโนโลยี”

โครงการ TH-AI Passport มีความย้อนแย้งสำคัญ คือ รัฐพยายามแสดงให้เห็นว่าโครงการเปิดกว้าง เช่น การเผยแพร่โมเดล Thai LLM-Base-1.0 บน Hugging Face ภายใต้สัญญาอนุญาต Apache 2.0

ในเชิงกฎหมาย สิ่งนี้ถือว่าเป็นความเปิดกว้างจริง เพราะผู้ใช้มีสิทธิ์นำโมเดลไปใช้ ดัดแปลง หรือพัฒนาต่อได้ตามเงื่อนไขของไลเซนส์

แต่ปัญหาคือ ในเชิงการใช้งานจริง คนส่วนใหญ่ยังใช้สิทธินั้นไม่ได้

เหตุผลคือ ระบบที่ส่งมอบ AI ให้ผู้ใช้จริงยังพึ่งพาโครงสร้างแบบ hosted model และ multi-vendor abstraction layer ซึ่งทำให้ผู้ใช้ทั่วไป นักพัฒนา SME หรือโรงเรียน ไม่สามารถนำโมเดลไปใช้งานในระบบของตนเองได้ง่าย

ดังนั้น แม้โมเดลจะเปิด แต่โครงสร้างการใช้งานยังไม่เปิดจริง

นักพัฒนาต้องการ API access และ benchmark สำหรับรันโมเดลภายในประเทศ
SME ต้องการคลัง use case และ workshop เฉพาะอุตสาหกรรม
นักเรียนต้องการตัวอย่างความสำเร็จที่ตรวจสอบได้
ประชาชนทั่วไปต้องการคู่มือใช้งานที่เข้าใจง่ายเป็นภาษาไทย

ทั้งหมดนี้สะท้อนว่า การมีโมเดลโอเพนซอร์สอย่างเดียวไม่พอ หากไม่มีเครื่องมือ workflow ตัวอย่าง และระบบสนับสนุนที่ทำให้คนใช้ได้จริง

อีกด้านหนึ่ง รัฐและผู้ให้บริการต่างชาติเปิดตัว Thai Data Sovereignty Dashboard v2.4 เพื่อแสดงข้อมูลเกี่ยวกับ data residency และ audit log แต่ dashboard นี้ทำหน้าที่เพียง “แสดงให้เห็น” ว่าข้อมูลอยู่ที่ไหน ไม่ได้แปลว่ารัฐสามารถควบคุมระบบ ประมวลผลโมเดลภายในประเทศ หรือย้ายข้อมูลได้จริงอย่างสมบูรณ์

พูดให้ชัดคือ อธิปไตยทางเทคโนโลยีไม่ใช่แค่การมี dashboard แต่ต้องหมายถึงความสามารถในการควบคุมระบบจริง ย้ายระบบได้จริง และพึ่งพาตนเองได้จริง

ในมุมของประชาชน ความย้อนแย้งนี้ทำให้ความเชื่อมั่นลดลง เพราะรัฐพูดเรื่องความเปิดกว้าง แต่ยังไม่สามารถตอบคำถามเรื่องการควบคุมข้อมูล การใช้ token การเชื่อมต่อกับ Thai LLM และการเข้าถึง API ได้ชัดเจน

แก่นของปัญหาจึงไม่ใช่การแจกสิทธิ์ใช้งาน AI แต่คือการสร้าง workflow ที่ประชาชนและองค์กรไทยใช้ได้จริง

# 3. ปัญหาการยอมรับใช้งาน: จากคนแห่สมัคร สู่คนเลิกใช้

ในช่วงแรก โครงการ TH-AI Passport อาจมีคนลงทะเบียนจำนวนมาก เพราะประชาชนสนใจ AI และต้องการทดลองใช้ แต่ปัญหาเกิดขึ้นหลังจากนั้นไม่นาน

ข้อมูลจำลองชี้ว่า หลังเปิดโครงการเพียงไม่กี่วัน ผู้ใช้เริ่มพบปัญหาหลายอย่าง เช่น token หมดเร็ว ไม่มี prompt template ภาษาไทย ไม่มีตัวอย่างการใช้งานสำหรับอาชีพหรืออุตสาหกรรมเฉพาะ และไม่มีระบบช่วยสอนให้ใช้ AI อย่างต่อเนื่อง

ผลคือ คนจำนวนมากสมัครแล้วไม่ได้ใช้งานต่อ

---

## 3.1 SME อยากใช้ AI แต่ไม่รู้จะใช้กับธุรกิจอย่างไร

กลุ่ม SME ไม่ได้ปฏิเสธ AI ตรงกันข้าม หลายคนต้องการใช้ AI เพื่อลดต้นทุน เพิ่มยอดขาย ทำคอนเทนต์ วิเคราะห์ข้อมูล หรือช่วยงานบัญชี

แต่ปัญหาคือ ระบบไม่ได้เตรียมเครื่องมือที่เหมาะกับงานจริงของ SME

สิ่งที่ SME ต้องการ ได้แก่

* template วิเคราะห์ยอดขาย
* workflow ช่วยทำบัญชีอัตโนมัติ
* ตัวช่วยเขียนคอนเทนต์ภาษาไทย
* ตัวอย่างการใช้ AI สำหรับธุรกิจค้าปลีก ร้านอาหาร บริการ หรือ e-commerce
* workshop เฉพาะอุตสาหกรรม
* คู่มือเริ่มต้นที่ไม่ซับซ้อน

ดังนั้น SME ไม่ได้ต้องการแค่สิทธิ์ใช้งาน AI Pro แต่ต้องการชุดเครื่องมือที่นำไปใช้ในธุรกิจได้ทันที

เมื่อไม่มีสิ่งเหล่านี้ การสมัครใช้งานจึงไม่เปลี่ยนเป็นการใช้งานจริง

---

## 3.2 นักเรียนสมัครเยอะ แต่ใช้งานต่อไม่มาก

นักเรียนเป็นกลุ่มที่มีความสนใจสูง เพราะ AI สามารถช่วยเรื่องการเรียน รายงาน การค้นคว้า และ portfolio ได้

แต่ระบบกลับไม่ได้เตรียมเส้นทางการใช้งานสำหรับนักเรียนอย่างชัดเจน เช่น

* ตัวอย่างรายงานที่ทำด้วย AI อย่างถูกต้อง
* prompt สำหรับช่วยวิเคราะห์ข้อมูล
* แนวทางใช้ AI โดยไม่ผิดจริยธรรมการเรียน
* ตัวอย่าง portfolio ที่ตรวจสอบได้
* กรณีศึกษาจากนักเรียนจริง
* คำแนะนำจากครูหรือสถาบันการศึกษา

นักเรียนไม่ได้ต้องการแค่เครื่องมือ AI แต่ต้องการหลักฐานว่า “ใช้แล้วได้ผลจริง” และ “ใช้แบบไหนถึงถูกต้อง”

หากไม่มีตัวอย่างความสำเร็จที่จับต้องได้ AI ก็จะกลายเป็นเพียงฟีเจอร์เสริมที่นักเรียนลองใช้ชั่วคราว แล้วเลิกใช้ในที่สุด

## 3.3 ประชาชนทั่วไปติดปัญหาตั้งแต่ขั้นตอนเริ่มต้น

ประชาชนทั่วไป โดยเฉพาะผู้สูงอายุ คนต่างจังหวัด และคนที่ไม่ถนัดเทคโนโลยี ต้องการขั้นตอนการสมัครที่ง่าย ชัดเจน และมีตัวอย่างการใช้งานใกล้ตัว

แต่ประสบการณ์ที่ได้รับกลับมีปัญหา เช่น

* การยืนยันตัวตนด้วย Thai ID ล้มเหลว
* onboarding เข้าใจยาก
* อินเทอร์เฟซเน้นภาษาอังกฤษ
* ไม่มีตัวอย่างการใช้งานในชีวิตประจำวัน
* ไม่มีเจ้าหน้าที่หรือช่องทางช่วยเหลือที่เข้าถึงง่าย
* ต้องพึ่งพาการสนับสนุนผ่านอินเทอร์เน็ตมากเกินไป

ตัวอย่าง use case ที่ประชาชนทั่วไปอาจต้องการ เช่น

* ใช้ AI ช่วยเขียนจดหมายขอบคุณผู้บริจาคให้วัด
* ใช้ AI ช่วยเขียนเอกสารเบิกค่ารักษาพยาบาล
* ใช้ AI ช่วยสรุปเอกสารราชการ
* ใช้ AI ช่วยเขียนข้อความขายของออนไลน์
* ใช้ AI ช่วยแปลหรืออธิบายเอกสารที่เข้าใจยาก

หากระบบไม่มีตัวอย่างเหล่านี้ คนทั่วไปจะรู้สึกว่า AI เป็นของไกลตัว และเลิกใช้หลังสมัครไม่นาน

# 4. ปัญหาใหญ่คือรัฐวัดผลจาก “ยอดสมัคร” มากกว่า “การใช้งานจริง”

หนึ่งในปัญหาสำคัญของโครงการคือ dashboard อาจแสดงตัวเลขที่ดูดี เช่น มีคนลงทะเบียน 5 ล้านคน แต่ไม่ได้เปิดเผยตัวเลขที่สำคัญกว่า เช่น

* มีผู้ใช้งานต่อเนื่องกี่คน
* อัตราการใช้งานหลัง 30 วันเหลือเท่าไร
* มีผู้ใช้ที่ทำ use case สำเร็จกี่เปอร์เซ็นต์
* SME ใช้แล้วลดต้นทุนได้จริงหรือไม่
* นักเรียนใช้แล้วเกิดผลงานจริงหรือไม่
* ประชาชนทั่วไปใช้ระบบได้โดยไม่ต้องมีคนช่วยหรือไม่

ถ้าวัดผลแค่ยอดลงทะเบียน โครงการจะดูประสบความสำเร็จในเชิงตัวเลข แต่ไม่ได้สะท้อนว่าประชาชนได้รับประโยชน์จริงหรือไม่

พูดอีกแบบคือ ยอดสมัครเป็นเพียง “ดอกไม้ไฟทางนโยบาย” แต่การใช้งานต่อเนื่องต่างหากที่เป็นหลักฐานของความสำเร็จจริง

# สรุปแบบอ่านง่าย

โครงการ TH-AI Passport มีเป้าหมายที่ดี คือทำให้คนไทยเข้าถึง AI ได้มากขึ้น แต่ในภาพจำลองนี้ โครงการเผชิญปัญหาใหญ่ 4 ด้าน

1. **การจัดซื้อยังไม่โปร่งใสพอ**
   เอกสารและโครงสร้างเทคนิคสำคัญยังไม่ถูกเปิดเผยอย่างชัดเจน

2. **ความเป็นส่วนตัวของข้อมูลยังไม่มีคำตอบ**
   ยังไม่ชัดเจนว่าข้อมูลผู้ใช้ถูกเก็บ ใช้ ตรวจสอบ และส่งต่ออย่างไร

3. **อธิปไตยทางเทคโนโลยียังไม่เกิดจริง**
   แม้มีโมเดลโอเพนซอร์ส แต่ระบบใช้งานจริงยังพึ่งพาผู้ให้บริการและโครงสร้างภายนอก

4. **ประชาชนสมัคร แต่ใช้งานต่อไม่มาก**
   เพราะไม่มี template ภาษาไทย ไม่มี use case ใกล้ตัว ไม่มี workflow เฉพาะกลุ่ม และไม่มีระบบสอนใช้งานที่ดีพอ

แก่นของรายงานนี้คือ TH-AI Passport จะสำเร็จไม่ได้ด้วยการแจกสิทธิ์ใช้งาน AI เพียงอย่างเดียว แต่ต้องสร้างระบบที่ทำให้ประชาชน SME นักเรียน และนักพัฒนา ใช้ AI ได้จริง เข้าใจง่าย ปลอดภัย โปร่งใส และเกิดประโยชน์ในชีวิตจริงหรือธุรกิจจริงได้อย่างต่อเนื่อง.
